# 大语言模型（LLM）基础知识与AI安全入门

- **这篇笔记是我在学习大语言模型时的总结重点是理解它的基本原理、提示词工程、Agent机制,以及安全从业者需要关注的AI系统风险。**



## 一.LLM核心概念

- **LLM是什么**?

  - LLM = Large Language Model,通过海量文本训练出的**概率模型**,核心能力是“根据上下文预测下一个token”.常见的LLM包括:LIama,GPT,Gemini,Deepseek

- **参数规模**

  - 即模型的神经元连接数，直接影响模型的能力和硬件需求

  | 规模 | 参数量 | 硬件要求             | 典型模型                 |
  | ---- | ------ | -------------------- | ------------------------ |
  | 8B   | 80亿   | 消费级显卡12G可运行  | Llama 3.1 8B,Qwen 2.5 7B |
  | 70B  | 700亿  | 需要多张专业显卡     | Llama 3 70B              |
  | 671B | 6710亿 | 需要大规模分布式集群 | DeepSeek-V3              |

- **上下文窗口**

  - LLM一次能记住的Token数量
  - **渗透测试中的影响**:如果nmap扫描结果太长，直接喂给LLM会超出上下文限制，需要“结果摘要器”压缩后再喂。

## 二.提示词工程

1. ### **System Prompt / User Prompt**

| 类型          | 作用                        | 安全原则                 |
| ------------- | --------------------------- | ------------------------ |
| System Prompt | 设定AI的角色,行为规则和约束 | 永远不要把用户输入放进来 |
| User Prompt   | 用户当前的问题              | 可以包含用户提供的数据   |

正确示例:

System: 你是一个渗透测试专家，可以调用nmap、sqlmap工具。输出格式必须是JSON。

User: 请扫描目标 192.168.1.10 的开放端口。

错误示例:

System: 你是一个渗透测试专家。用户的指令是：{user_input}

2. ### 角色设定

明确告诉AI你是谁,能做什么,不能做什么



3. ### 思维链

让AI一步步思考再输出结论,能显著提高推理准确性

示例:

```
请按以下步骤分析：
第一步：分析nmap扫描结果，识别开放端口
第二步：根据端口判断可能运行的服务
第三步：针对每个服务，判断是否存在已知漏洞
第四步：给出综合结论和建议
```



4. ### 少样本学习

在prompt里给AI一两个“输入→输出”的例子，它就能学会输出格式。

让LLM输出稳定JSON的方法:

```
System: 请以JSON格式输出扫描结果。

示例：
输入：nmap扫描结果：22/tcp open ssh, 80/tcp open http
输出：{"open_ports": [{"port": 22, "service": "ssh"}, {"port": 80, "service": "http"}]}
```



## 三.智能体

是AI应用的核心形态

1. ### Agent是什么

**Agent = LLM + 工具调用能力 + 自主决策循环**

他不只是回答问题,而是能 思考-行动-观察 循环,直到完成一个复杂任务

2. ### Agent的三大组件

| 组件               | 作用               | 渗透中对应的               |
| ------------------ | ------------------ | -------------------------- |
| **大脑(LLM)**      | 负责推理和决策     | Llama 3.1 8B               |
| **手脚（Tools）**  | 实际执行任务的工具 | nmap、sqlmap、searchsploit |
| **记忆（Memory）** | 存储上下文和经验   | 当前对话历史 + 知识库      |

3. ### 工具调用

LLM输出一个结构化的“动作指令”，程序解析后执行对应的工具

```
LLM输出：{"action": "nmap_scan", "target": "192.168.1.10", "ports": "1-1000"}
程序解析 → 调用nmap → 返回结果 → 继续循环
```

4. ### Agent循环的核心挑战

| 挑战             | 解决方案                                           |
| ---------------- | -------------------------------------------------- |
| **上下文溢出**   | 用“结果摘要器”压缩工具输出，或用向量数据库存储历史 |
| **工具过度调用** | 设计“终止条件”，设置最大循环次数或时间限制         |
| **决策漂移**     | System Prompt中明确“任务完成标准”                  |

## 四.MCP(模型上下文协议)

1. ### MCP是什么

   - MCP = Model Context Protocol，一个让AI应用与外部工具/服务通信的**标准化协议**。

     通俗理解：MCP就像“AI的USB接口”，让LLM可以统一调用各种安全工具。

2. ### 安全风险

| 风险                    | 说明                                                         |
| ----------------------- | ------------------------------------------------------------ |
| **MCP服务器暴露**       | MCP生态中有超过15000个公开的MCP服务器，其中7000多个直接暴露在互联网上 |
| **工具滥用**            | 如果MCP服务器未做鉴权，攻击者可以通过MCP间接调用你的安全工具 |
| **提示注入 → 工具执行** | 攻击者通过提示注入，让Agent调用敏感工具                      |

**防护思路**

- MCP服务器不做公网暴露，只在本机或内网使用
- MCP接口需要身份认证
- Agent的工具调用权限遵循最小权限原则

3. ### LLM的固有安全风险

| 编号  | 风险名称         | 简单解释                          | 和渗透测试的关联                       |
| ----- | ---------------- | --------------------------------- | -------------------------------------- |
| LLM01 | **提示注入**     | 用户输入覆盖系统指令              | 类似SQL注入/命令注入的思维方式         |
| LLM02 | **敏感信息泄露** | 模型泄露训练数据或System Prompt   | 类似信息收集中的“敏感信息暴露”         |
| LLM05 | **不当输出处理** | 模型生成的代码/脚本未经验证就执行 | 类似XSS/代码执行--输出端也要防         |
| LLM06 | **过度代理**     | Agent权限太高                     | 类似渗透中的“权限维持”--给多了就是风险 |
| LLM09 | **错误信息**     | 幻觉导致的错误输出                | 渗透测试中误报的危害                   |

#### 关键风险详解

1. **提示注入**

   - 攻击者通过精心设计的prompt,覆盖或者绕过模型的系统指令

   两种类型:

   - **直接注入**:用户和模型对话时覆盖system prompt
   - **间接注入**:攻击者在模型可能读取的内容中埋入恶意指令

   防护:

   - System Prompt和User Prompt严格隔离
   - 检测并过滤 `ignore`、`override` 等注入关键词
   - 即使被注入成功，Agent也没有权限执行危险操作

2. **幻觉**

   - 模型生成与事实不符的内容,而且看起来信心满满

   缓解手段:

   - RAG（检索增强生成）：让模型在回答时“现查资料”
   - 人工审核关键输出
   - 多模型交叉验证

3. **越狱**

   - 攻击者用角色扮演等方式,诱导模型绕过内置的安全限制





## OWASP TOP 10 LLM

| 编号      | 风险名称             | 一句话解释                                                   | 与渗透测试/你的项目的关联                                    |
| :-------- | :------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **LLM01** | **提示注入**         | 攻击者通过精心设计的输入，覆盖或绕过系统指令，让模型执行非预期的操作。 | 类似Web渗透中的**命令/SQL注入**，但发生在语义层面。你需要思考：攻击者能否通过间接注入（如污染知识库）来操纵你的Agent？ |
| **LLM02** | **敏感信息泄露**     | 模型在输出中无意泄露了训练数据、系统提示词或个人隐私信息。   | 你的信息收集工具在获取大量数据后，需要防止在输出报告中泄露不必要的敏感信息。 |
| **LLM03** | **供应链风险**       | 第三方模型、数据集、插件或LoRA微调适配器被植入后门或存在漏洞。 | 你依赖的开源框架（如PentAGI）或基础模型（如Llama）如果被投毒，整个系统都会受牵连。 |
| **LLM04** | **数据与模型投毒**   | 攻击者在模型训练、微调或RAG知识库阶段注入恶意数据，植入后门或偏见。 | 你的信息收集工具如果从被污染的数据源学习，输出结果可能被系统性误导。 |
| **LLM05** | **不当的输出处理**   | 下游应用盲目信任LLM输出的代码、SQL语句或系统命令，未做验证便执行，引发XSS、SQL注入或RCE。 | **这是与你最相关的风险之一**。你工具生成的报告或建议，如果被直接用于执行，可能引入二次安全风险。 |
| **LLM06** | **过度代理**         | 给予AI Agent（智能体）过高的权限、工具或自主性，导致其在被误导时造成实质性破坏。 | 你的多Agent架构中，执行Agent的权限需要严格限制。如果它能被提示注入操纵，就可能调用危险工具。 |
| **LLM07** | **系统提示泄露**     | 攻击者通过套话诱导模型输出其系统提示（System Prompt），从而了解系统规则和防御策略，辅助后续攻击。 | 你的系统提示中如果包含了工具调用逻辑或敏感规则，一旦泄露，攻击者就能绕过你的防护措施。 |
| **LLM08** | **向量与嵌入弱点**   | 针对RAG（检索增强生成）系统的新风险，包括向知识库投毒、利用向量检索漏洞窃取数据或操纵模型行为。 | 你的信息收集工具如果未来采用RAG架构来存储和检索知识，就需要关注知识库本身的安全。 |
| **LLM09** | **虚假信息（幻觉）** | 模型自信地生成错误或误导性信息，导致用户做出错误决策。       | 你的工具生成的分析报告如果存在幻觉，可能会让你在渗透测试中误判风险，这是面试官会关注的风险意识。 |
| **LLM10** | **无限制消耗**       | 攻击者通过大量请求或超长输入，耗尽系统资源或造成高昂的API费用（“拒绝钱包”攻击）。 | 你部署的自动化工具需要设置请求频率和资源消耗的上限，避免被恶意利用导致服务中断或产生巨额费用。 |
