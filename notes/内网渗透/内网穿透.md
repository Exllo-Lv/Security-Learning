# 基础概念
内⽹穿透，也即 NAT 穿透，进⾏ NAT 穿透是为了使具有某⼀个特定源 IP 地址和源端⼝号的数据包不被 NAT 设备屏蔽⽽正确路由到内⽹主机

### 为什么要进行内网穿透？
webshell不是交互式的，因为webshell管理工具本身是基于http协议去做的，http协议是无状态的，不利于我们做后渗透

CS这样的工具则是通过心跳包做连接的，心跳包会比webshell这种连接方式好，但是当我们要用这台机器去打另一台机器的时候只能基于这些工具上面自有的命令或攻击方式去做后渗透，也会有一定局限性，

所以我们想要 用外网DMZ区我们已经拿到权限的这台机器去访问内部网络的时候，我们要把DMZ区的这个机器当作我们的流量代理或者跳板，让我们能够把自己机器的流量带入到实际目标的内部网络中，以便于我们更好的完成后渗透工作

### 哪些机器需要做内网穿透?
我们在DMZ区拿到了一台机器，这台机器在能够访问外网的同时也能访问内网，这台机器就是我们需要做内网穿透的机器

很多的目标集团公司内网中会有DMZ区（主要放对外的web服务）、办公区、生产区、核心区，这些区域之间他们的网络可能不会强打通，比如说DMZ区的机器想要访问办公区的网络，一般来说在常规的集团内网结构里面只会有DMZ区的一两台机器能够访问到办公区，因为默认情况下DMZ区的机器只需要对外提供服务就好了，不需要去跟内部的网络做通信，只需要内部的网络对他做交互或管理就行了，假设我们现在在DMZ区找到了一台机器，这台机器能够联通到办公区，如果这台机器能出网的话，我们可以对这台机器也做内网穿透



### 不同穿透模式试用场景?
内网穿透分不同模式，常见的有正向代理和反向代理，不同模式适用于不同的场景

```plain
正向代理适用的场景：DMZ区对外提供服务（如web服务）的机器不能主动往外发包，也就是不能主动出网，只能对外提供服务，此时可以用正向代理，简单来说就是不出网用正向代理，
```

```plain
反向代理适用的场景：出网的时候使用反向代理
```

上图中绿色的防火墙（最外面的防火墙）会把DMZ区的对外提供服务的机器（假设是web服务）映射到公网的ip地址（防火墙一般是有公网ip地址的），攻击者通过访问DNS服务器来访问到防火墙的公网ip,

防火墙可以有几种不同的设定：一种是允许对外提供服务的机器主动访问外网，也可以主动提供服务，这样的好打一点；还有一种模式就是对外提供服务的机器只有一个端口的映射，假设这个对外提供服务的机器内网的IP是10.10.xx.xx,公网ip地址是162.xx.xx.xx,防火墙在做映射的时候会把内网的ip地址80端口映射到公网ip的80端口，此时攻击者访问的时候访问的是162.xx.xx.xx：80

像我们上面的第二种设定，对外提供服务的机器就没办法主动出网，他只能做映射

正向代理是我们访问162.xx.xx.xx：80，他把流量转到办公网的机器上面，

反向代理是对外提供服务的机器会主动连接公网上我们攻击者服务器



如果防火墙对我们正向连接的端口做限制了，那我们就连接不了了，所以一般来说我们在做正向的时候，我们会把他正向的流量嵌入到他对外提供服务的端口上面，最常见的就是我们在做正向的时候会去挂载到他对外提供服务的web服务上，至于没有web服务的情况下就要看他防火墙具体的策略了

正向的局限性就是，多数情况下我们用正向的时候是这台机器的web服务能够支持我们做正向代理，如果web服务不支持我们就做不了正向，就拿我们常用的做正向代理的工具reG那个来说，他需要往我们web服务器上传一个正向代理的脚本，这个脚本能够解析，能够正常执行，然后我们才能通过这个脚本做正向连接，不能解析的话我们的正向就做不了了





# 正向代理和反向代理
### 什么是代理
代理其实就是⼀个中介，A和B本来可以直连，中间插⼊⼀个C，C就是中介。

刚开始的时候，代理多数是帮助内⽹client访问外⽹server⽤的

后来出现了反向代理，"反向"这个词在这⼉的意思其实是指⽅向相反，即代理将来⾃外⽹客户端的请求转发到内⽹服务器，从外到内

### 正向代理
正向代理类似⼀个跳板机，代理访问外部资源

⽐如我们国内访问⾕歌，直接访问访问不到，我们可以通过⼀个正向代理服务器，请求发到代理服，代理服务器能够访问⾕歌，这样由代理去⾕歌取到返回数据，再返回给我们，这样我们就能访问⾕歌了


![](https://exllo-lv-github-notes-picture-1452130078.cos.ap-nanjing.myqcloud.com/img/1778982445653-9899d045-f323-4643-a27e-61d66badad0b.png)

我们在获得外⽹服务器的⼀定权限后发现这台服务器可以直接或者间接的访问内⽹ ，此时渗透测试进⼊后渗透阶段，⼀般情况下，内⽹中的其他机器是不允许来⾃外⽹机器的访问的。这时候，我们可以将这台外⽹服务器设置成为代理，使得我们⾃⼰的攻击机可以直接访问与操作内⽹中其他机器。（就是我们访问不到内网服务，但是我们的流量给正向代理服务器，正向代理服务器给带内网服务器，内网服务器的数据给到正向代理服务器，正向代理服务器再把数据给到我们的电脑）

按照代理协议在TCP/IP协议栈中所处的位置，可以将正向代理分为HTTP代理和socks代理。HTTP代理技术要求所有代理流量按照HTTP协议进⾏传输，使⽤socks代理技术则可以在直接在⽹络层传输代理数据包。HTTP代理⼯作在应⽤层，socks代理⼯作在⽹络层，所以socks代理技术⽐HTTP代理快很多。

### 反向代理
反向代理（Reverse Proxy）实际运⾏⽅式是指以代理服务器来接受internet上的连接请求，然后将请求转发给内部⽹络上的服务器，并将从服务器上得到的结果返回给internet上请求连接的客户端，此时代理服务器对外就表现为⼀个服务器



![](https://exllo-lv-github-notes-picture-1452130078.cos.ap-nanjing.myqcloud.com/img/1778982459329-e6c5d2d4-b3a8-4e1e-8192-502496f88507.png)

假设我DMZ区拿到一台机器（就上图反向代理的服务器，假设他有内网ip也有外网ip），此时我想要访问内部网络的时候，我可以让DMZ区的机器主动连接客户端

### Socks协议
不管做正向还是反向用的最多的就是Socks协议

什么是Socks协议？？

中⽂意思：防⽕墙安全会话转换协议，⼯作在OSI参考模型的第5层（会话层）。  
它是⼀种可以穿透防⽕墙的协议，很多场景都会⽤到。⽐如Fan墙，你们懂得~~  
因为Socks介于传输层与表示层之间，使⽤TCP协议传输数据，因⽽不提供如传递ICMP信息之类的⽹络层相关服务。  
⽬前有两个版本：SOCKS4和SOCKS5  
SOCKS4⽀持TELNET、FTP、HTTP等TCP协议；  
SOCKS5⽀持TCP与UDP，并⽀持安全认证⽅案。

socks5可以加密，socks4没有加密方案

Ps: Socks不⽀持ICMP，不能使⽤ping命令



# Ngrok内⽹穿透
如果以我们开发⼈员的机⼦作为服务端，有开发环境，⼜有tomcat，数据库等等，如果能够直接将机⼦这个内⽹的IP映射到公⽹多好啊。这就是我们将要介绍的内⽹穿透利器，⽽且是免费的！！

ngrok 是⼀个⽤ go 语⾔编写的反向代理，通过在公共的端点和本地运⾏的 Web 服务器之间建⽴⼀个安全的通道。ngrok 可捕获和分析所有通道上的流量，便于后期分析和重放。

ngrok 是⼀个反向代理，通过在公共的端点和本地运⾏的 Web 服务器之间建⽴⼀个安全的通道。Ngrok是外国发布的⼀个开源项⽬，也有⼀个公开的运⾏服务，但被国内墙掉了。国内⼀个机构建⽴并运营了⼀个ngrok运⾏服务，虽然有时不太稳定，但对于调试⼈员来说是⾜够的。地址：[http://www.ngrok.cc/](http://www.ngrok.cc/) 。其⽀持HTTP的80端⼝和TCP端⼝转发。下⾯介绍使⽤⽅法。微信开发调试即需要⼀个公⽹IP/域名和80端⼝。

使用方法：

1） 注册登陆ngrok平台，申请⼀个ngrok.cc下的⼆级域名，并在基本信息⾥⾯获得token值。

2） 下载ngrok客户端并运⾏，输⼊token值即可。⾮常简单。这样本机就可以作为⼀个web服务器啦。

cmd切换到安装的ngrok.exe⽬录下，命令： ngrok -authtoken 密钥 -subdomain ⼆级域名 端⼝

3） 接着访问即可



花生壳、Ngrok转发TCP流量是会员功能，他俩官网上入门介绍里通过绑定域名实现内网穿透的都是走的HTTP/HTTPS流量



# frp内⽹穿透
简单地说，frp就是⼀个反向代理软件，它体积轻量但功能很强⼤，可以使处于内⽹或防⽕墙后的设备对外界提供服务，它⽀持HTTP、TCP、UDP等众多协议。我们今天仅讨论TCP和UDP相关的内容。（不⽀持socks）  
搭建⼀个完整的frp服务链，我们需要

1. VPS⼀台（也可以是具有公⽹IP的实体机）
2. 访问⽬标设备（就是你最终要访问的设备）
3. 简单的Linux基础（会⽤cp等⼏个简单命令即可）

### 服务端设置
SSH连接到VPS之后运⾏如下命令查看处理器架构，根据架构下载不同版本的frp

```plain
arch
```

查看结果，如果是“X86_64“即可选择”amd64”，  
运⾏如下命令，根据架构不同，选择相应版本并进⾏下载

```plain
wget https://github.com/fatedier/frp/releases/download/v0.22.0/frp_0.22.0_linux_amd64.tar.gz
```

然后解压：

```plain
tar -zxvf frp_0.22.0_linux_amd64.tar.gz
```



```plain
cd
```

⽂件夹改个名，⽅便使⽤

```plain
cp -r frp_0.22.0_linux_amd64 frp
```

把解压出来的⽂件夹复制到你想要的⽬录下，为了⽅便我直接放在⽤户⽬录下了，进⼊该⽬录

```plain
cd frp
```

我们只需要关注如下⼏个⽂件

+ frps
+ frps.ini
+ frpc
+ frpc.ini

前两个⽂件（s结尾代表server）分别是服务端程序和服务端配置⽂件，后两个⽂件（c结尾代表client）分别是客户端程序和客户端配置⽂件。  
因为我们正在配置服务端，可以删除客户端的两个⽂件

```plain
rm frpc
rm frpc.ini
```

然后修改frps.ini⽂件  
这个⽂件应有如下格式

```plain
[common]
bind_port = 7000		//客户端和服务端连接的端⼝,一会配客户端的时候端口也要设置成7000
dashboard_port = 7500
token = 12345678
dashboard_user = admin
dashboard_pwd = admin
vhost_http_port = 10080
vhost_https_port = 10443
```

如果没有必要，端⼝均可使⽤默认值，token、user和password项请⾃⾏设置。

+ “bind_port”表示⽤于客户端和服务端连接的端⼝，这个端⼝号我们之后在配置客户端的时候要⽤到。
+ “dashboard_port”是服务端仪表板的端⼝，若使⽤7500端⼝，在配置完成服务启动后可以通过浏览器访问x.x.x.x:7500 （其中x.x.x.x为VPS的IP）查看frp服务运⾏信息。
+ “token”是⽤于客户端和服务端连接的⼝令，请⾃⾏设置并记录，稍后会⽤到。
+ “dashboard_user”和“dashboard_pwd”表示打开仪表板⻚⾯登录的⽤户名和密码，⾃⾏设置即可。
+ “vhost_http_port”和“vhost_https_port”⽤于反向代理HTTP主机时使⽤，本⽂不涉及HTTP协议，因⽽照抄或者删除这两条均可。

编辑完成后保存。  
之后我们就可以运⾏frps的服务端了

```plain
./frps -c frps.ini
```

如果看到屏幕输出这样⼀段内容，即表示运⾏正常，如果出现错误提示，请检查上⾯的步骤。

其中common字段下的三项即为服务端的设置。

+ “server_addr”为服务端IP地址，填⼊即可。
+ “server_port”为服务器端⼝，填⼊你设置的端⼝号即可，如果未改变就是7000
+ “token”是你在服务器上设置的连接⼝令，原样填⼊即可。

上⾯frpc.ini的rdp、smb字段都是⾃⼰定义的规则，⾃定义端⼝对应时格式如下。

+ “[xxx]”表示⼀个规则名称，⾃⼰定义，便于查询即可。
+ “type”表示转发的协议类型，有TCP和UDP等选项可以选择，如有需要请⾃⾏查询frp⼿册。[https://www.cnblogs.com/sanduzxcvbnm/p/8508741.html](https://www.cnblogs.com/sanduzxcvbnm/p/8508741.html)
+ “local_port”是本地应⽤的端⼝号，按照实际应⽤⼯作在本机的端⼝号填写即可。
+ “remote_port”是该条规则在服务端开放的端⼝号，⾃⼰填写并记录即可。

RDP，即Remote Desktop 远程桌⾯，Windows的RDP默认端⼝是3389，协议为TCP，建议使⽤frp远程连接前，在局域⽹中测试好，能够成功连接后再使⽤frp穿透连接。

SMB，即Windows⽂件共享所使⽤的协议，默认端⼝号445，协议TCP，本条规则可实现远程⽂件访问。

