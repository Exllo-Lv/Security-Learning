## 文件上传漏洞

### 漏洞描述

文件上传漏洞是指由于程序员再开发时，未对用户上传的文件进行严格的验证和过滤，而导致用户可以上传可执行的动态脚本文件。  

如常见的的头像上传,文件附件上传,媒体上传等,允许用户上传文件,如果过滤不严格,恶意用户上传了可执行的文件或者脚本,就会导致网站被其控制甚至会使服务器沦陷,以至于引发恶意的网络安全事件



### 漏洞原理

用户通过界面的上传功能上传了一个可执行的脚本文件,而WEB的系统并未对其进行检测或者检测的逻辑存在缺陷

通常来说文件上传没有问题,若WEB未对用户上传的文件进行有效的审查,若存在恶意用户对其上传一句话木马,从而实现控制WEB网站的目的

### 攻击方法

寻找测试网站文件上传的模块,常分布在:头像上传,文件附件,文章编辑,资料认证,编辑器上传等位置

抓取上传文件的数据包进行测试,上传后缀为asp .php.jsp.aspx等格式的文件,查看响应包,判断能否直接上传 ,不能上传再测试图片



### 文件上传绕过

#### 绕过前端JS检测上传

在文件上传时,用户选择文件时或者提交时,有些网站会对前端文件名进行校验,一般检测后缀是否为上传的格式,如果上传的格式不被允许,则提示不被允许

可通过

- 禁用浏览器Js
- 使用Burp拦截请求后更改后缀.再在服务端吧后缀改回来



#### 服务端验证绕过

1. **绕过文件扩展名检查**

服务器禁止上传`asp`、`php`、`jsp`等危险后缀，但使用的是黑名单机制

- 大小写:	`PhP`、`pHp`、`.htACCESS`等。如果服务器没有统一转为小写再比较，就可能绕过。
- 双写:可以构造`pphphp`。
- 利用其他可行后缀:黑名单往往不全。除了`php`，还可以尝试`php3`、`php5`、`phtml`；ASP系可以试试`aspx`、`ashx`、`asmx`；JSP系有`jspx`。如果服务器把这些后缀也当作脚本执行，就中招了。

2. **绕过MIME类型验证**

服务器检查 `Content-Type`请求头

- 使用抓包工具修改`Content-Type`

3. **绕过文件内容/文件头验证(图片马)**

服务器用`getimagesize()`或检查文件头来确保文件是真正的图片。

- 制作图片马
  - 在Windows命令行使用`copy`命令，将一张正常图片和PHP木马文件合并。`copy shell.jpg/b + shell.php/a logo.jpg`（`/b`表示二进制，`/a` [Clash Verge.lnk](..\..\..\..\Public\Desktop\Clash Verge.lnk) 表示ASCII）。
  - 新生成的logo.jpg ,文件头是图片信息,但是文件末尾包含了PHP代码

#### 服务器和代码逻辑缺陷

1. **"00"截断(空字符截断)**

- **场景**：PHP版本低于5.3.4，且文件路径通过`$_GET`或`$_POST`拼接，未做过滤。
- **原理**：在C语言和早期PHP中，`%00`（十六进制的空字符）会被解释为字符串的**结束标志**。
- **攻击**：构造文件名`shell.php%00.jpg`。程序保存文件时，`%00`后面的`.jpg`被当作注释“吞掉”，最终在服务器上生成一个`shell.php`文件。这个漏洞在较新的PHP版本中已修复。

2. **条件竞争**

- **场景**：网站的处理逻辑是“先上传，再检查，发现恶意则删除”。
- **原理**：在“检查并删除”这个间隙，攻击者通过多线程并发访问刚刚上传的临时文件，抢先执行其中的恶意代码。
- **攻击**：
  1. 上传一个文件`shell.php`，其内容是`<?php file_put_contents('shell.php', '<?php eval(...);?>'); ?>`。
  2. 当文件刚落地，网站还没来及删除它时，攻击者的脚本**瞬间发起大量请求**访问`http://target.com/uploads/shell.php`。
  3. 只要有一个请求在文件被删除前触发了PHP代码，这个木马就会永久生成在服务器上。

### **服务器解析漏洞导致文件上传**

- **IIS6.x**

1. **目录解析**：该版本默认将 *.asp/ 目录下的所有文件当成Asp解析

2. **畸形文件解析**：该版本默认将*.asp;.jpg 此种格式的文件名，当成Asp解析。服务器默认不解析 ; 号及其后面的内容，相当于截断。iis除了会将asp解析成脚本执行文件之外，还会将 cer、cdx、asa 扩展名解析成asp；

3. **PUT漏洞**：IIS Server 在 Web 服务扩展中开启了 WebDAV ，配置了可以写入的权限，造成任意文件上传。

- **IIS7.x**

1. 在Fast-CGl运行模式下，在一个文件路径 /xx.jpg 后面加上/.php会将 /xx.jpg/.php 解析为php文件。



- **Nginx解析漏洞**

1. 该漏洞是由于Nginx中php配置不当而造成的，与Nginx版本无关，但在高版本的php中，由于 security.limit_extensions 的引入，使得该漏洞难以被成功利用。在PHP的配置文件中有一个关键的选项php.ini cgi.fix_pathinfo=1 默认是开启的，在一个文件/xx.jpg后面加上/.php会将 /xx.jpg/xx.php 解析为 php 文件。

2. CVE-2013-4547：在上传图片时，修改其16进制编码可使其绕过策略，导致解析为php。当Nginx得到一个用户请求时，首先对url进行解析，进行正则匹配，如果匹配到以.php后缀结尾的文件名，会将请求的PHP文件交给PHP-CGI去解析。当php-fpm处理时碰到0x00截断，则图片会当作php文件解析，触发图片马.



- **Apache解析漏洞**

1. **未知扩展名解析漏洞**：在Apache1.x/2.x中 Apache默认一个文件可以有多个以点分割的后缀，解析文件的规则是从右到左开始判断解析,如果后缀名为不可识别文件解析,就再往左判断。直到识别到合法后缀才进行解析。如 1.php.aaa.xxx如果都不认识将会暴露源码

2. **AddHandler解析漏洞**：在apache配置不当的时候就会造成apache解析漏洞

3. **CVE-2017-15717**：Apache HTTPD是一款HTTP服务器，它可以通过mod_php来运行PHP网页。其2.4.0~2.4.29版本中存在一个解析漏洞，在解析PHP时，1.php0x0a 将被按照PHP后缀 进行解析，导致绕过一些服务器的安全策略。

## 文件上传防御的思路和挖洞的思路**



**测试思路:**

1.**侦察和信息收集**

- **识别上传入口**,编辑器插件,API接口,导入导出
- **判断使用的技术栈**
  - 通过URL后缀（`.php`、`.asp`、`.jsp`）、响应头（`X-Powered-By`）、Cookie（`JSESSIONID`）等**指纹信息**，判断后端语言和中间件版本。

2.**分层测试与绕过**

- **前端限制层**
  - 上传PHP文件,若弹窗提示文件类型不允许,那说明是前端校验
  - 抓包,发送到Repeter模块,先上传合法的jpg,抓到包后吧文件名修改成shell.php,`content-type`保持不变发送
- **服务端校验层**

| 校验类型         | 观察什么                                                     | 绕过思路（攻击载荷示例）                                     |
| :--------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **文件扩展名**   | 上传`shell.php`被拦截，上传`shell.jpg`成功。                 | **黑名单绕过**：`shell.PHP`（大小写）、`shell.phtml`（其他后缀）、`shell.pphphp`（双写）。 **白名单绕过**：重点关注**解析漏洞**和**文件包含**，详见第三阶段。 |
| **Content-Type** | 上传成功/失败时，对比请求包的`Content-Type`字段。            | 直接修改为`image/jpeg`、`image/png`等图片类型。              |
| **文件头**       | 服务器可能会校验前几个字节。                                 | 制作**图片木马**：`copy 1.jpg/b + shell.php/a shell.jpg`。 或者直接在`shell.php`文件头部添加`GIF89a`，变成`GIF89a<?php phpinfo();?>`。 |
| **文件内容**     | 触发WAF（Web应用防火墙）拦截，或返回“文件包含恶意代码”。     | **编码混淆**：`<?php`可以变形为`<script language="php">`、`<%@ page import="java.lang.*" %>`等，或用`base64`、`ROT13`编码后配合`eval`执行。 |
| **二次渲染**     | 上传图片后，服务器会重新生成缩略图，可能导致你的代码被删掉。 | **攻击点转向**：放弃在图片中插入代码，转而寻找**文件包含漏洞**，通过上传一个包含恶意路径的文本文件来利用。 |
