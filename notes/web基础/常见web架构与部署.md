# 常见web架构与部署

## 单体架构

- **定义**：所有代码（前端、后端、业务逻辑）打包成一个整体部署。
- **部署特征**：单台服务器 + 单一数据库。
- **渗透视角**：
  - **文件上传绕过**：如果代码和上传目录在同一台服务器，拿到Webshell就等于拿到服务器权限。
  - **本地文件包含（LFI）**：直接读取 `/etc/passwd` 或源码。
  - **端口暴露**：通常只需要扫 `80/443` 和 `22/3389`。

## 前后端分离架构

- **定义**：静态前端（Vue/React）与后端API（Java/Go/PHP）分开部署。
- **部署特征**：
  - **前端**：通常放在Nginx静态目录或CDN/OSS上。
  - **后端**：运行在内部端口（如 `8080`），通过Nginx反向代理对外暴露 `/api/` 路径。

### 反向代理和应用服务器协作

反向代理（如Nginx、Apache）和应用服务器（如Tomcat）是Web系统中分工明确、密切合作的两种核心组件。简单来说，**反向代理是系统的“门卫”，负责接收所有外部请求并分发；而应用服务器是“后端团队”，负责执行具体的业务逻辑**。

| 对比维度     | 反向代理(Nginx/Apache)                          | 应用服务器(Tomcat)                                      |
| ------------ | ----------------------------------------------- | ------------------------------------------------------- |
| **核心职责** | 请求路由,负载均衡,处理静态资源,SSL终止,安全防护 | 运行业务代码,处理动态请求(JSP,Servlet),执行核心业务逻辑 |
| **擅长领域** | 高并发连接管理                                  | 复杂业务计算                                            |
| **典型位置** | 系统边缘,作为内部应用集群的统一入口和屏障       | 系统内部,在内网隐藏在反向代理之后                       |
| **协议支持** | HTTP,HTTPS,也可通过模块支持其他协议(AJP)        | 主要支持HTTP/HTTPS.也支持AJP协议以优化与代理的通信      |

## 关键组件攻击面

### 反向代理(Nginx/Apache)

反向代理是用户请求的第一道关卡,攻击原通常不直接硬碰WAF规则,而是利用代理和后端应用对HTTP协议理解的差异

1. **Header注入**：`X-Forwarded-For` 伪造，绕过CDN获取真实IP或触发SQL注入（WAF误判）。
2. **HTTP请求走私 (HTTP Desync)**：利用 `Content-Length` 与 `Transfer-Encoding: chunked` 不一致，绕过前端WAF，直接向后端发送恶意Payload。
3. **静态目录遍历**：尝试访问 `./css/../../` 下载 `.git` 或 `.env` 文件。
4. **缓存投毒**
5. **SSL/TLS配置缺陷**
6. **日志注入**

#### 攻击流程

1. curl -I 目标 -> 看 Server: nginx 确认版本
2. 目录扫描时加入路径穿越字典：/../../、/../、/%2e%2e/、//admin//
3. Burp发送请求，修改 Transfer-Encoding 头，观察响应差异判断是否可走私
4. 测试 X-Forwarded-For: 127.0.0.1 能否访问 /admin
5. 访问 /.git/config、/.env、/docker-compose.yml 等敏感文件

### 应用服务器(Tomcat/Jetty)

1. **Manager 后台**：Tomcat 默认 `/manager/html`，尝试弱口令爆破 (tomcat/tomcat) -> 上传WAR包拿Shell。
2. **弱口令**
3. **Tomcat  Session反序列化**:使用了FileStore时,通过上传恶意序列化文件触发反序列化
4. **Spring Boot**的/actuator暴露,重点关注/env(环境变量) /heapdump(内存快照)/congigprops(配置属性)
5. **文件上传**
6. **Session管理缺陷**

#### 攻击流程

1. 访问根路径，观察返回头（Server: Apache-Coyote/1.1 是Tomcat）
2. 拼接 /manager、/actuator、/env、/swagger-ui.html 看是否暴露
3. 随便访问一个不存在的路径，看报错页面获取版本和路径
4. 用Burp Spider爬取所有JS文件，搜索 api/、/v1/、/admin/ 等接口
5. 检查 Cookie 的 Set-Cookie 属性（是否带 HttpOnly）
6. 检查是否有文件上传点，尝试上传 .jsp 或 .html 绕过

### 数据库(MySQL/Redis/MongoDB)

1. **未授权访问**
2. **弱口令爆破**
3. **文件读写**
4. UDF**提权**
5. **注入**
6. **内网横移**
7. **备份文件泄露**

#### **攻击流程**

1. 看到3306/6379/27017端口开放，先用nc/telnet试探
2. Redis：redis-cli连接，用 info 查看服务器信息，keys * 看是否有敏感数据
3. MySQL：尝试空密码连接，若成功则查询 user、password 表；若失败则爆破
4. 连接成功后查看权限（MySQL的 user 表、Redis的 CONFIG GET *）
5. 判断能否写文件或执行命令
6. 若数据库内网可达，尝试用数据库作为跳板扫描其他内网主机

## 防御思考

1. **非ROOT运行**:所有服务用低权限用户,防止提权拿shadow和passwd
2. **日志集中化**:便于溯源和关联攻击IP
3. **网络隔离:**数据库只允许应用服务器的IP访问,不允许公网连接
